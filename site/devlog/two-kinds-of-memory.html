Two kinds of memory: what you search, and what you keep

The Neonmem team · Jun 22, 2026 · ← Back to Dev Corner

Every developer who has paired with an AI agent for more than a week knows the ghost. Monday it understood your whole architecture. Tuesday it's a polite stranger asking what the project does. So you re-explain — every morning, the same context, paying the tokens each time. The usual answer is "use a bigger context window," but that's treating amnesia with flashcards: you re-teach the same facts forever, and the model still can't tell this decision from one you reversed three weeks ago. Context is short-term memory. What's missing is the long-term kind.

A memory product has one job: when the agent needs to know something about your project, it should be there — and it should be right. The fastest way to lose a user's trust is to confidently invent something. So 0.9.7 was about two things: importing your project the way it actually exists, and making recall grounded — answers that come from your sources, not the model's imagination.

The bug that reframed the release

Our test corpus is a large enterprise system, and one question kept embarrassing us. Ask "what is ARC?" and the agent would confidently answer "Automatic Reference Counting" — the textbook expansion. Wrong. In these documents ARC is a specific in-house platform. The agent wasn't reading the memory we'd given it; it was pattern-matching on its training and producing a plausible, authoritative, wrong answer.

And a memory that's occasionally wrong is worse than no memory at all. With no memory you stay skeptical and go check. With a memory that's wrong five percent of the time, you relax — and that's exactly when it burns you. Trust is binary. So the real specification was never "remember more." It was: answer from the user's sources, or honestly say you don't know. Everything else serves that.

Your project isn't one kind of thing

When we watched real imports, an obvious truth kept getting in the way: the stuff you want an agent to remember comes in two very different shapes, and they want opposite treatment.

  • Documents and code are reference material. You don't want them "summarised" into a handful of memories — you want all of it, searchable, so any detail can be recalled on demand. That's a knowledge pool.
  • Agent conversations are decisions. A chat with your assistant isn't reference material — it's where you decided things, hit dead-ends, and set rules. You don't want the whole transcript; you want the handful of conclusions worth keeping.

The old importer treated both the same — it flattened everything into one pile. 0.9.7 splits them.

Folders & files → a searchable pool

Point Neonmem at your docs and code and it vectorises the whole corpus into a lossless, deduplicated facts pool — the same fact stated three ways becomes one fact, with every source kept. Nothing is thrown away or paraphrased; the original passages are retrievable by meaning or by wording. Ask "what is ARC?" and the answer is lifted from your documents — your team's own provisioning platform — not a plausible-sounding guess.

You can tag an import with the topics you care about. Tag Site API and Neonmem mints one clean, canonical memory for it linked back to the source — even when your docs never write the words "Site API" verbatim, as long as they clearly describe it. If the corpus genuinely has nothing on a tag, it's left out rather than faked.

Agent chats → memories you keep

The new Agent chat importer reads a Claude (or other agent) transcript and pulls out only what's worth remembering: the decisions, the dead-ends, the rules. "Decided: we'll adopt the Site API as the integration surface." "The first embedding approach failed because the vectors landed in a different space." "Rule: never co-author commits." A decision is stored as a decision; a dead-end stays a warning.

It deliberately throws away the noise — the "I read the file…", the "please check…", the back-and-forth — and it deduplicates every kept memory through the same facts layer, so re-importing a conversation never doubles up. One canonical statement, linked to its context.

And if a chat references a file on disk, Neonmem follows the link: that file is pulled into the pool, with a memory that points back to it. The conversation and the artefacts it touched arrive together.

The counterintuitive part: a good memory is mostly about forgetting

This is the idea we find most interesting, and it's borrowed straight from neuroscience. Map a memory system onto the brain and you get a tidy picture: a large, distributed store of what you know, and a process that replays the day's experience into long-term form. Most "AI memory" builds those two and stops.

But there's a third part people forget to model, and it's the one that matters most — the regulator that runs homeostasis, and in particular the sleep cycle that decides what gets consolidated and what gets cleared. Biological memory stays sharp precisely because it forgets aggressively. A memory that only ever accumulates becomes a junk drawer: slower to search, full of stale and contradicted facts, increasingly confidently wrong. The magic isn't storage — it's the discipline of deciding what not to keep.

So Neonmem has its own version of that regulator. A memory is one clean, canonical statement linked to its source — never a wall of raw text. From a conversation it keeps the signal and discards the narration. And while the cartridge "sleeps", an offline consolidation pass merges duplicates and lets go of facts nothing references anymore, so the store stays lean instead of bloating. We spent as much effort on what the memory refuses to keep as on what it stores — and that restraint is the difference between a demo and something you trust.

Grounded, and local

Underneath both paths is a new offline embedding engine — a compact model that lives inside the app and runs entirely on your CPU, giving database-class retrieval quality without the weight. Every prompt walks memory from the fast, instinctive layers to the slow, deliberate ones, and answers from what's actually there — or abstains. If the answer is in your sources, it surfaces; if it isn't, the agent says so instead of inventing one.

It all runs on your machine. Partly that's a privacy stance — but mostly it's that a memory of your work, your decisions and your dead-ends is yours, and shouldn't be a row in someone else's database.
