The Locked-Down Deploy

Shipping onto a locked-down network.

Step 1 · Frame the problem · OBSERVATION

The engineer frames the environment. Agent takes in the constraints. Neonmem stores the first picture of the network.

Step 2 · Frame the problem · OBSERVATION

The engineer adds the awkward details. Agent notes them. Neonmem keeps each constraint beside the project.

Step 3 · Frame the problem · QUESTION

The engineer raises an open question. Agent holds it open. Neonmem records the question for the debate.

Step 4 · Set the plan · PLAN

The engineer sets the trajectory before touching anything. Agent records it as a PLAN — the forward-looking node (indigo) Neonmem keeps near the centre while it's active, so the goal and next steps survive across sessions.

Step 5 · Design debate · HYPOTHESIS

The engineer floats an approach. Agent treats it as a hunch to test. Neonmem files it as a hypothesis.

Step 6 · Design debate · DEBATE

The engineer opens a debate. Agent lays out both options. Neonmem starts a debate thread.

Step 7 · Design debate · ARGUMENT

The engineer weighs speed against compliance. Agent balances them. Neonmem links each argument to the debate.

Step 8 · Design debate · RESOLUTION

The engineer decides. Agent commits to the compliant path. Neonmem stores the resolution permanently.

Step 9 · Make it work · ACTION

The engineer asks for the change. Agent configures the route. Neonmem logs the action.

Step 10 · Make it work · CONCLUSION

The engineer confirms success. Agent validates the route. Neonmem records it as the working path.

Step 11 · Hit a wall · DEAD_END

The engineer flags a silent failure. Agent marks it. Neonmem stores a permanent dead-end so no one re-trips the firewall.

Step 12 · Hit a wall · RULE

The engineer sets a rule. Agent adopts it. Neonmem pins it near the core.

Step 13 · Instant knowing · PROCEDURE

The engineer records the safe procedure. Agent captures the steps. Neonmem stores the how-to.

Step 14 · Instant knowing · REFLEX

The engineer sets the guiding principle. Agent internalizes it. Neonmem keeps it as a reflex.

Step 15 · Lean on memory · RECALL

The engineer asks for the route. Agent answers from memory. Neonmem surfaces the working path and the procedure.

Step 16 · Lean on memory · DEAD_END_CHECK

The engineer almost reopens a bad idea. Agent warns it's a recorded dead-end. Neonmem flags the failure and why.

Step 17 · Lean on memory · TIMELINE

The engineer asks what happened today. Agent replays it in order. Neonmem returns it by time, not a guess.

Step 18 · Let it sleep · DREAM

The engineer wraps up. Agent consolidates. Neonmem merges the chatter and keeps the constraints, dead-ends and rules.

Step 19 · Feed it the runbook · OBSERVATION

The engineer feeds in the runbook and config. Neonmem gathers the endpoints, ports and config values as facts, and skips the prose.

Step 20 · Know the whole · SKELETON

The engineer asks for the big picture. Agent reads it from Neonmem's comprehension layer — a small self-built model that clusters every node into subsystems. The brain colours by subsystem and Neonmem returns the deploy's skeleton, not a re-reading of the runbook.

Step 21 · Know the whole · RESUME

The engineer tests a cold start. Agent answers with the goal, the binding constraints and the resume cursor — the exact thread left open — without scrolling back. The glowing coherence core hands a fresh session the whole picture at once.
