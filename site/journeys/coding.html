The Labyrinth

Building a small game with the agent

Step 1 · Frame the project · OBSERVATION

The developer frames the project. Agent takes it in and starts a fresh thread. Neonmem stores the project's first memory.

Step 2 · Frame the project · OBSERVATION

The developer describes how rooms work. Agent notes the model. Neonmem keeps the detail beside the project.

Step 3 · Frame the project · QUESTION

The developer raises an open question. Agent holds it open instead of guessing. Neonmem records the question so it isn't lost.

Step 4 · Set the plan · PLAN

The developer sets the direction up front. Agent records it as a PLAN — Neonmem's one forward-looking node (the indigo one). It resists fading while active, so a fresh session always knows where we're headed, not just where we've been.

Step 5 · Design debate · HYPOTHESIS

The developer floats an idea. Agent treats it as a hunch to test. Neonmem files it as a hypothesis.

Step 6 · Design debate · DEBATE

The developer opens a real debate. Agent lays out both sides. Neonmem starts a debate thread to hold the tension.

Step 7 · Design debate · ARGUMENT

The developer gives points for and against. Agent weighs them. Neonmem links each argument to the debate.

Step 8 · Design debate · RESOLUTION

The developer decides. Agent commits to the call. Neonmem stores the resolution where it won't fade.

Step 9 · Build it · ACTION

The developer asks for code. Agent writes the first version. Neonmem logs the action taken.

Step 10 · Build it · CONCLUSION

The developer confirms it works. Agent validates the approach. Neonmem records the conclusion as settled.

Step 11 · Hit a wall · DEAD_END

The developer flags a past failure. Agent marks it clearly. Neonmem stores a permanent dead-end so it's never tried again.

Step 12 · Hit a wall · RULE

The developer turns the lesson into a rule. Agent adopts it. Neonmem pins the rule close to the core.

Step 13 · Instant knowing · PROCEDURE

The developer records a repeatable procedure. Agent captures the steps. Neonmem stores the how-to for next time.

Step 14 · Instant knowing · REFLEX

The developer sets a standing principle. Agent internalizes it. Neonmem keeps it as a reflex at the centre.

Step 15 · Lean on memory · RECALL

The developer asks why. Agent answers from memory, not a guess. Neonmem surfaces the original decision.

Step 16 · Lean on memory · DEAD_END_CHECK

The developer almost repeats a mistake. Agent warns it's a known dead-end. Neonmem flags the recorded failure.

Step 17 · Lean on memory · TIMELINE

The developer asks what happened today. Agent replays the day in order. Neonmem returns it by time, not from imagination.

Step 18 · Lean on memory · RECALL

The developer asks for the standing rule. Agent recalls the reflex instantly. Neonmem hands it back word for word.

Step 19 · Let it sleep · DREAM

The developer wraps up. Agent runs a consolidation. Neonmem merges the noise and keeps the decisions, dead-ends and rules.

Step 20 · Feed it the spec · OBSERVATION

The developer hands Neonmem the spec and code. It gathers the endpoints, classes, methods and the call graph as connected facts — and skips the filler.

Step 21 · Know the whole · SKELETON

The developer asks for the big picture. Agent reads it straight from Neonmem's comprehension layer — a small self-built model that clusters every node into subsystems. The brain colours by subsystem and Neonmem hands back the project's skeleton, not a re-reading of the files.

Step 22 · Know the whole · RESUME

The developer tests a cold start. Agent answers with the point, the goal and the resume cursor — the exact thread we left open — without scrolling back. The glowing coherence core hands a fresh session the whole understanding at once.
